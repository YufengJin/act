### Known limitations

1. The cosmetic `RuntimeError: cannot schedule new futures after shutdown` and `free(): invalid pointer` lines at process exit are a well-known `dm_control` teardown bug; they appear after the last useful line of output and do not affect any test result. The same bug propagates a non-zero (rc=134) exit from short-lived helper scripts (e.g. `capture_spec.py`) — for spec capture this run, the canonical interface is hand-curated and matches the verified registry, so the rc=134 from auto-introspection is non-blocking.
2. dm_control auto-introspection of `env.action_spec` returns the **16-D raw MuJoCo actuator surface** (each finger has mirrored pos/neg actuators). The **14-D policy interface** (6 joints + 1 normalized gripper per arm) is what `sim_env.before_step()` accepts and what the policy server advertises. The `.nautilus/benchmark-spec.json` records the 14-D interface (matches the verified registry).
3. The OSMesa env-var fix lives in `docker/docker-compose.headless.yaml`. If you switch to `docker-compose.x11.yaml` for GUI work, you may want to set `MUJOCO_GL=glfw` (or leave it unset and let dm_control auto-detect the X11 stack) instead — the OSMesa setting is a software-rendering fallback chosen for headless reliability, not GPU performance.
