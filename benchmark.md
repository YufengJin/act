# act — Benchmark Guide

Generated by `benchmark-generator` skill. English-only by contract; regenerated on every re-run.

## About

ACT (Action Chunking with Transformers) is an imitation-learning framework for fine-grained bimanual manipulation. A Transformer-based policy predicts a chunk of future joint-position actions from RGB observations and proprioception, allowing low-frequency inference to drive high-frequency control. This repository ships two MuJoCo simulation benchmarks — Transfer Cube and Bimanual Insertion — together with the training, recording, evaluation and visualization tooling that accompanies the original ACT paper. Each task has a scripted-demonstration variant and a human-demonstration variant, primarily intended as a behavior-cloning testbed.

| Field | Value |

| Upstream paper / project | Zhao et al., *Learning Fine-Grained Bimanual Manipulation with Low-Cost Hardware* (RSS 2023) — original repo by Tony Zhao |

## Tasks

The benchmark exposes **4** tasks. Each row below lists the task identifier used by `--task_name`, the underlying MuJoCo XML, the maximum task reward, and a short description.

| `--task_name` | XML | max_reward | Description |
|---|---|---|---|
| `sim_transfer_cube_scripted` | `bimanual_viperx_transfer_cube.xml` | 4 | Right arm picks the cube off the table and hands it off to the left arm, which receives the cube. Trajectories are produced by a hand-coded scripted policy. |
| `sim_transfer_cube_human` | `bimanual_viperx_transfer_cube.xml` | 4 | Same Transfer-Cube physics as above, but the demonstration source is human teleoperation (used to study scripted-vs-human BC gaps). |
| `sim_insertion_scripted` | `bimanual_viperx_insertion.xml` | 4 | Left arm grasps and holds a socket; the right arm picks up a peg and inserts it into the socket — a precise bimanual contact-rich task. Demonstrations from a scripted policy. |
| `sim_insertion_human` | `bimanual_viperx_insertion.xml` | 4 | Same Bimanual-Insertion task, demonstrations from human teleop. |

The `_scripted` and `_human` task variants share environment dynamics and only differ in the demonstration source used for behavior cloning.

## Action / Observation spaces

| Field | Value |
|-------|-------|
| Action dim | `14` |
| Action semantics | Absolute joint positions for both arms plus normalized gripper commands. Layout: `[left_arm_qpos(6), left_gripper(1), right_arm_qpos(6), right_gripper(1)]`. Gripper values are normalized: `0` → fully closed, `1` → fully open. |
| Observation type | `dict` with `qpos` (14-D joint positions), `qvel` (14-D joint velocities), `env_state` (object pose), and `images` — a sub-dict with three RGB camera streams (`top`, `angle`, `vis`/`front_close`), each `480 × 640 × 3 uint8`. |
| Control frequency | 50 Hz (MP4 outputs are written at `fps=50`) |
| Reward signal | Stage-based sparse integer reward in `{0, 1, 2, 3, 4}`. For Transfer Cube: 1 = right gripper touches cube, 2 = cube lifted, 3 = left gripper touches cube, 4 = cube transferred (left holds, off table). For Insertion: 1 = both grippers hold their objects, 2 = both objects grasped (off table), 3 = peg touches socket, 4 = peg fully inserted. Episode-level success = `max_reward_in_episode >= 4`. |

## How to use

The skill produced two receipt files at the repo root — `history.md` (what the skill did) and this file (what the benchmark *is*). `install.md` (how to build) is rendered separately by `env-generator`. The flow below assumes you have already followed `install.md` and have a running container.

### 1. Enter the container

```bash
docker compose -f docker/docker-compose.headless.yaml up -d
docker exec -it act-headless bash
```

### 2. Pick a task

```bash
# List available tasks (one per line)
python -c "from constants import SIM_TASK_CONFIGS; print('\n'.join(SIM_TASK_CONFIGS.keys()))"
```

Any identifier from the task table above is a valid value for `--task_name`.

### 3. Stand up a policy server

For a smoke test, use the companion random-policy server (ships with the image):

```bash
python tests/test_random_policy_server.py --port 8765
```

For real evaluation, replace it with any server that speaks the `policy_websocket` protocol and advertises `action_dim=14` in its metadata handshake — the client's guard in `scripts/run_eval.py` and `scripts/run_demo.py` enforces the match and raises `ValueError` on mismatch before any rollout starts.

Each step the client sends an observation dict that includes the mandatory `__meta__` envelope (`{v: 1, benchmark: "act", task, task_description, phase}`). Servers that don't care about meta can ignore the key — the random-policy companion does exactly that.

### 4. Run a single-episode demo

```bash
python scripts/run_demo.py --policy_server_addr localhost:8765 \
    --task_name sim_transfer_cube_scripted --n-steps 400
```

`run_demo.py` is the "show me one rollout" entry point — per-step rewards are printed, and (with `--save_video`) one MP4 of the `top` camera stream is saved per run under `logs/demo/<task>--<timestamp>/`. A `demo.log` mirrors stdout in the same directory.

### 5. Run multi-episode eval

```bash
python scripts/run_eval.py --policy_server_addr localhost:8765 \
    --task_name sim_transfer_cube_scripted --n-episodes 10 --save_video
```

Prints success rate (= fraction of rollouts that reached `max_reward = 4`) and mean return across `--n-episodes` rollouts. Per-episode MP4s are saved to `logs/eval/<task>--<timestamp>/episode_{idx:03d}_success={bool}.mp4`, and a text run log is written to `logs/eval/<task>--<timestamp>/eval.log`.

### 6. Train / collect demonstrations

For behavior cloning on this benchmark you will typically:

1. Collect or download demonstration episodes (see `record_sim_episodes.py` in the repo root for the scripted-policy collection harness; demos for all four tasks are mirrored at https://drive.google.com/drive/folders/1gPR03v05S1xiInoVJn7G7VJ9pDCnxq9O).
2. Train your policy off-line against those demos (`imitate_episodes.py`).
3. Wrap the trained policy in a `policy_websocket`-speaking server (mirrors `tests/test_random_policy_server.py`).
4. Point `scripts/run_eval.py --policy_server_addr host:port` at that server to measure success rate.

The skill does **not** generate a training script for IL benchmarks — BC training code is benchmark-specific and upstream usually ships its own. See `imitate_episodes.py` (top of the repo) for the official ACT training entry point.

## Troubleshooting

- `ConnectionRefused` on `run_eval.py` -> no policy server on the configured port. Start `tests/test_random_policy_server.py` first (IL) or check the server process logs. Default port is **8765** (matches `policy_websocket` library default).
- `ValueError: server action_dim=... != expected 14` -> the policy server advertises a different action dim. Fix the server, or re-generate with `BENCHMARK_CATEGORY=IL` + the correct `ACTION_DIM`.
- `mujoco.FatalError: an OpenGL platform library has not been loaded` on first `env.reset()` -> the headless compose profile must export `MUJOCO_GL=osmesa` and `PYOPENGL_PLATFORM=osmesa` (already set in `docker/docker-compose.headless.yaml`). If you launch the container by hand without compose, pass `-e MUJOCO_GL=osmesa -e PYOPENGL_PLATFORM=osmesa` to `docker run`.
- Task name not recognised -> check the task table above; identifiers are case-sensitive snake_case.
- GUI apps fail with `cannot open display` -> you are on the headless compose profile. Switch to `docker-compose.x11.yaml` after `docker compose down` (and either unset `MUJOCO_GL` or set it to `glfw`).
- Cosmetic `free(): invalid pointer` / `cannot schedule new futures after shutdown` lines at process exit -> known `dm_control` teardown bug; harmless, appears after the last useful line of output.
