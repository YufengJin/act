# act — Install Recipe

Minimum commands to reproduce this benchmark container from a fresh clone. Generated by `benchmark-env-generator` skill (sub-skill of `env-generator`).

## Host prerequisites

- Docker (>= 24) and the Compose v2 plugin
- NVIDIA driver + NVIDIA Container Toolkit with a GPU visible to `nvidia-smi`

## Build the image

```bash
docker compose -f docker/docker-compose.headless.yaml build
```

Base image: `nvidia/cuda:11.8.0-cudnn8-runtime-ubuntu20.04`
Python: `3.9` (managed by `uv`)

`act` and the `detr` workspace member are pre-synced into `/opt/venv` at build time and re-installed editable from the bind-mounted source on every container start (handled by `docker/entrypoint.sh`).

## Bring the container up

```bash
docker compose -f docker/docker-compose.headless.yaml up -d
docker exec -it act_container bash
```

For an X11 / GUI session (swap the compose profile — container name is shared, only one may run):

```bash
xhost +local:docker
docker compose -f docker/docker-compose.x11.yaml up -d
```

## First-run sanity checks (inside the container)

```bash
nvidia-smi
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
python -c "import sim_env; print(sim_env.__file__)"
```

The last line must resolve under `/workspace/` — that proves the mounted source is what the interpreter imports (editable install succeeded). `act` does not expose a top-level package; the canonical proof-of-import is `import sim_env` (the bimanual MuJoCo env factory module).

## Category-specific smoke run

### IL: WebSocket random-policy loop

The generated image ships with a companion server so the stack is end-to-end runnable without a trained policy:

```bash
# Start the random-policy server (advertises action_dim=14)
docker exec -d act_container python tests/test_random_policy_server.py --port 8000

# Wait for the port (runtime image has no ss / nc — use Python sockets)
docker exec act_container python -c "import socket, time
for _ in range(20):
    try: socket.create_connection(('127.0.0.1', 8000), timeout=1).close(); print('ready'); break
    except OSError: time.sleep(0.5)"

# Single-episode demo (records one MP4 under demo_log/)
docker exec act_container python scripts/run_demo.py \
    --task_name sim_transfer_cube_scripted \
    --policy_server_addr localhost:8000 --num_resets 1

# One-rollout eval (writes eval.log + per-episode MP4 under eval_logs/)
docker exec act_container python scripts/run_eval.py \
    --task_name sim_transfer_cube_scripted \
    --policy_server_addr localhost:8000 --num_rollouts 1

# Cleanup
docker exec act_container pkill -f test_random_policy_server
```

Both clients perform a metadata handshake against the server's advertised `action_dim` and refuse to start if it does not equal the env's expected `14`. The random policy never solves the task — its only job is to exercise the eval / demo wiring end-to-end.

To plug in a real policy, replace `tests/test_random_policy_server.py` with any server that speaks the `policy_websocket` protocol and advertises `action_dim=14` in its metadata.

## Troubleshooting

- `ConnectionRefused` on `run_eval.py` / `run_demo.py` → no policy server on the configured port. Start `tests/test_random_policy_server.py` first.
- `ValueError: server action_dim=... != expected 14` → the policy server advertises a different action dim. Fix the server.
- `cannot open display` from any GUI flag → you are on the headless compose profile. Switch to `docker-compose.x11.yaml` after `docker compose down`.
- Cosmetic `free(): invalid pointer` / `cannot schedule new futures after shutdown` at process exit → known dm_control teardown bug; appears after the script's last useful line and does not affect results.
